Build a Fraud-Triage AI Crew in AMP — and Watch It Think — Tutorial¶

In ~30 minutes, from your browser and with no code, you build a two-agent fraud-triage crew, run it on a transaction you choose, watch both agents reason step-by-step in a live trace, and deploy it as a real service in your own workspace.

Step 1 — Describe what you want (Crew Studio)¶

A fraud analyst's day is a firehose of flagged transactions. A model gives you a risk score — say 0.87 — but a score isn't a decision. Someone still has to weigh it against what actually happened, call it fraud or not, and write it up so the next person can act. That judgment work, at volume, is exactly what a crew of AI agents is good at — if you can trust it. Let's build that crew, no code.

Open Crew Studio from the left rail (under Build). You'll land on "What would you like to build today?" with a Describe your automation box. Paste this in and hit the blue → (Build automation):

Triage a suspicious financial transaction for fraud. Given a transaction record and its machine-learning risk score, first have a senior fraud analyst weigh the risk score against the transaction attributes (amount, merchant category, device, location, account balance, time of day) and return a FRAUD or APPROVED verdict with evidence-based reasoning. Then have a fraud QA specialist review that verdict, catch any missed red flags, and produce the final verdict plus a short case narrative for escalation.

💡 What just happened. You didn't pick a template — you described the job in plain English. Studio reads that and proposes a crew: a team of AI agents, each with a role, that work in sequence. That's the core CrewAI idea, and you'll watch it take shape.

✅ Checkpoint: Studio opens a project editor with a Studio Chat panel and a Canvas.

Step 2 — Steer the build (keep it self-contained)¶

Studio now works out how to build your crew. It may do one of two things — either ask you a couple of clarifying questions first, or go straight to proposing a plan. Both are normal; don't worry about which you get.

If it asks questions, answer with the self-contained option each time so everything runs right here in your browser — nothing to connect: - When it asks where the transaction comes from → choose Plain text input (manually pasted) (not Gmail/Zendesk). - When it asks where the output should go → choose Display in the crew output only (not a Sheet or email).

If it goes straight to a plan, great — just read it and move to the next step.

💡 Why this matters. Those self-contained choices are the difference between a demo that needs your company's Slack, inbox, and admin — and one that just works for you, right now. "Plain text in, result on screen" is the whole no-setup story. (Studio is an AI — it won't ask the exact same thing every time, so steer by that principle rather than by a fixed script.)

✅ Checkpoint: Studio shows a plan and asks you to confirm it.

Step 3 — Confirm the plan → your crew appears¶

Read the plan Studio proposes — you'll recognize your two agents in it — then click Confirm plan. (Nothing is built until you confirm; if you tried to Run now you'd get "No tasks found.")

Watch the Canvas build in front of you:

  • Senior Fraud Analyst — weighs the risk score against the transaction details and calls it FRAUD or APPROVED, with reasoning.
  • a QA / quality-assurance agent — a second agent that reviews the first one's verdict, checks for missed red flags, and writes the final case record. (Studio names it something like Fraud QA Specialist or Fraud Quality Assurance Specialist — the exact wording varies.)

They run sequentially: the analyst decides, then the QA agent double-checks. (A suggestion tip may appear in the chat panel — leave it open; it often holds the Run Automation button you'll use next, so dismissing it just hides that button.) Studio also picks a project name on its own (e.g. FraudTriage-Crew) — that varies run to run and doesn't matter; what matters is the two agents wired analyst → QA.

💡 The idea worth keeping. You didn't ask for a checker — but good fraud process always has one. Notice AMP gave your crew a QA agent that audits the analyst before anything is final. That "second set of eyes" is why you can trust the output later.

✅ Checkpoint: The Canvas shows two agents wired in order — a fraud analyst → a QA agent (Studio's auto-generated project name varies and doesn't matter).

Step 4 — Run it on your transaction¶

Start a run by clicking the blue Run button (top right) — that reliably opens the run form. (You may also see a black Run Automation button inside the Studio Chat panel; it does the same thing. Don't switch to the "Run" tab expecting a button — it can just show a "waiting for events…" spinner.) Because you chose plain-text input, there's nothing to connect — AMP gives you a simple form (titled Run parameters) with one box per input the plan generated; each box is labeled with the raw field name (e.g. transaction_id).

Your form may not match this list exactly. Studio designs the inputs from your description, so the exact field names/count can vary run to run. Don't worry — just fill each box you see with a sensible value. Use the suspicious transaction below as your guide; map each value to the closest field, and if your build has a field not listed here, make up a fictional value that fits.

What it represents Example value
transaction id TXN-4471-0093
amount 4999.00
merchant category Electronics
merchant name QuikByte Wholesale (unrecognized merchant)
device mobile (new device, first seen today)
location Lagos, NG (IP region)
account balance 512.30
time of day 03:17 AM
ML risk score 0.87
account history notes Account opened 6 days ago; no prior transaction above $200; billing address is US.

Fill every box the form shows, then click Execute.

💡 This is "your own data." Change the amount, drop the risk score to 0.15, make the merchant a familiar one — and re-run. The crew reasons about what you gave it, not a canned script. Try to make it flip to APPROVED.

✅ Checkpoint: The Run/execution view shows Execution started and a live Timeline.

Step 5 — Watch it think (the trace)¶

This is the part a slide deck can't fake. The Timeline shows each agent as it runs — each one makes an LLM call (the AI model doing the actual reasoning), and you can watch it happen:

  • Senior Fraud Analyst → Analyze Transaction Risk: Started → LLM call (~6s) → Completed
  • Fraud QA Specialist → QA Review & Final Verdict: Started → LLM call (~4s) → Completed

Click any step to open Event details (with a Raw Data tab) — the exact instruction the agent got and what it produced. Click the QA specialist's Completed step — then scroll past the Description (the agent's instructions) to the Output section below, which is the actual final result. On a real run of the transaction above it returned:

Final Verdict: FRAUD · Recommended Action: Block & Investigate QA Findings — amount $4,999 far exceeds the $512.30 balance · new device first seen today · unrecognized merchant · location (Lagos, NG) vs US billing · 03:17 AM unusual time. Missed red flags: none. Overrides: none — the analyst aligns with the evidence. Plus a plain-English case narrative for escalation.

💡 Why this is the whole point. You can see which agent decided what, on which evidence, and how long it took. When your fraud-ops lead asks "why did it block this?", the answer isn't "the AI said so" — it's a trace you can open. That's the difference between a laptop script and a platform you'd put near production.

✅ Checkpoint: You've read the final FRAUD verdict and seen both agents' steps in the trace.

Step 6 — Deploy it as a real service¶

A crew that only runs in the editor is a prototype. Make it real: click Deploy (top right). In Ready to go live, keep the suggested name (whatever Studio generated, e.g. …-v1), optionally add a release note, and click Deploy.

Go to Automations in the left rail — your crew is there, Source Studio. It first shows Building Images for Crew…, then goes Online, with its own endpoint (a web address other systems can call) and an access token. This can take a few minutes — if it looks stuck on Building Images for Crew, refresh the page and it'll flip to Online. Identify the one you just created (it's the newest entry — don't confuse it with an older, similarly-named crew someone made before). It now runs without your editor open — a service you (or a system you connect later) can call.

💡 What you just crossed. From "I built something in a tool" to "I have a running service in my workspace." Same crew, now deployed — and everything you run against it shows up in the same kind of trace you just read.

What you did: in ~30 minutes, no code — you described a fraud-triage crew, built it (analyst + a QA checker), ran it on a transaction you chose, read the trace to see exactly how it decided, and deployed it as a live service in your own workspace. Every claim it made, you could verify.

Take it further: feed it a borderline case and watch the QA agent overturn the analyst · deploy a second version and compare · swap the scenario for your own domain (the same shape works for KYC review, claims triage, or pharmacovigilance intake).

✅ Checkpoint: Your newly-deployed crew (the newest Studio-source entry) is listed under Automations and goes Online.
